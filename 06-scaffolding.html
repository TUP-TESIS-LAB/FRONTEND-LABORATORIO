Principios

  • Standalone components. Sin NgModules; cada feature exporta sus rutas (*.routes.ts).
  • Lazy por feature. Cada módulo activable carga bajo demanda con loadChildren y un guard que valida si está activo para el tenant.
  • White-label vía CSS vars. Los colores del tenant viven en tokens.css como variables; Tailwind las consume desde su theme.extend.
  • Multi-tenant. Un interceptor inyecta el tenant_id en cada request; un resolver carga la config del tenant antes de montar el shell.
  • ModuleRegistry. Servicio en core/tenant que expone qué módulos están activos. El sidebar y las rutas se construyen a partir de él.

Activación de módulos

1 · Tenant config

El backend devuelve { modules: ["portal", "turnos", "financiero"] } al iniciar sesión.

2 · ModuleRegistry

Mantiene la lista en memoria; provee isActive(key) y activeRoutes().

3 · Sidebar dinámico

Renderiza items core siempre y items activables solo si isActive; los inactivos pueden mostrarse con candado (mockup 4).

4 · Guard por ruta

moduleActiveGuard('turnos') evita acceso directo por URL si el módulo está apagado.

app.routes.ts (esquema)

export const APP_ROUTES = [
  { path: '', canActivate: [authGuard],
    loadComponent: () => import('./layout/shell'),
    resolve: { tenant: tenantResolver },
    children: [
      // CORE
      { path: 'empresa',    loadChildren: () => import('./features/empresa/empresa.routes') },
      { path: 'sucursales', loadChildren: () => import('./features/sucursales/sucursales.routes') },
      { path: 'analitica',  loadChildren: () => import('./features/analitica/analitica.routes') },

      // ACTIVABLES — guard adicional
      { path: 'portal',     canMatch: [moduleActive('portal')],     loadChildren: () => import('./features/portal/portal.routes') },
      { path: 'turnos',     canMatch: [moduleActive('turnos')],     loadChildren: () => import('./features/turnos/turnos.routes') },
      { path: 'financiero', canMatch: [moduleActive('financiero')], loadChildren: () => import('./features/financiero/financiero.routes') },
      { path: 'medicos',    canMatch: [moduleActive('medicos')],    loadChildren: () => import('./features/medicos/medicos.routes') },
      { path: 'stock',      canMatch: [moduleActive('stock')],      loadChildren: () => import('./features/stock/stock.routes') },
    ],
  },
  { path: 'portal-publico', loadChildren: () => import('./features/portal/landing.routes') },
  { path: 'login',         loadComponent:  () => import('./core/auth/login') },
  { path: 'admin',         canActivate:    [rootGuard],
                            loadChildren:  () => import('./features/saas-admin/admin.routes') },
];

Convenciones

  • Naming. Carpetas en kebab-case; archivos como feature.component.ts, feature.service.ts, feature.routes.ts.
  • Path aliases. @core/*, @shared/*, @features/*, @layout/* en tsconfig.json.
  • Estado. Signals + servicios. Si crece: NgRx Signal Store solo en features que lo necesiten.
  • Tests. *.spec.ts al lado del archivo. E2E con Playwright en e2e/.
  • i18n. ngx-translate o @angular/localize con archivos en assets/i18n/.
